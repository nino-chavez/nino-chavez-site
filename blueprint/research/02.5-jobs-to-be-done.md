# 02.5 — Jobs-to-be-Done

The methodology gap. Blueprint Stages 1–2 produced research + prescription;
Stage 3 was supposed to be a design brief but I wrote one as a positioning
artifact, not as a job spec. The result: design that aesthetically expressed
positioning without serving any concrete user job. Rally HQ didn't have this
gap because the product function dictated jobs directly; personal sites
don't get jobs for free.

This document fills the gap: per-surface, per-persona, per-time-budget
acceptance criteria. The forge-site portfolio-brand archetype + forge-brand
generated tokens + forge-signal voice rules consume these as design
constraints. The brief stops being an aesthetic statement and becomes a
testable spec.

## Discipline

Each job:
- **Surface**: which page
- **Persona**: who is doing this (cross-ref `personas/_index.md`)
- **Trigger**: how they arrived
- **Time budget**: hard deadline in seconds before they bounce
- **Job**: the verb the visitor needs to execute
- **Acceptance**: testable rules ("within N seconds, sees X / can do Y")
- **Anti-acceptance**: what design choices fail this job

The visual/IA brief that follows MUST satisfy every acceptance line. Designs
that violate anti-acceptance get rejected at Stage 4 review regardless of
how good they look.

## J1 — Peer Architect verifies the practice is real

| | |
|---|---|
| **Surface** | `/` |
| **Persona** | P1 Peer Architect (`personas/peer-architect.md`) |
| **Trigger** | Arrival from Signal Dispatch post (most likely) OR direct via referral |
| **Time budget** | 90 seconds total; 5-second hook |
| **Job** | Verify the practice exists before bookmarking, RSS-subscribing, or returning |

**Acceptance**:
1. Within 5 seconds of landing: sees ≥3 named shipped products with live URLs visible without scrolling
2. Within 30 seconds: can open one product OR open the Signal Dispatch archive in ≤1 click
3. Within 90 seconds: has either opened a product tab, opened the blog, copied the URL, or RSS-subscribed
4. Capability claim is parseable in ≤4 seconds (≤12 words, ≤2 lines, no hero-display scale that prevents scanning)

**Anti-acceptance** (designs that fail this job):
- Capability statement at display/hero scale that fills above-the-fold and prevents product visibility
- Photo anchor placeholder that beats receipts for visual weight
- Receipt list buried below the fold
- Single-large-image hero that makes the page feel like a brand-poster, not a working-architect's home

## J2 — Prospective Client decides whether to schedule a conversation

| | |
|---|---|
| **Surface** | `/` |
| **Persona** | P2 Prospective Client (`personas/prospective-client.md`) |
| **Trigger** | Arrival via LinkedIn referral OR colleague recommendation |
| **Time budget** | 60 seconds total; 10-second capability check |
| **Job** | Decide if Nino can solve their specific commerce / architecture problem; reach the booking flow if yes |

**Acceptance**:
1. Within 10 seconds: sees capability statement in vocabulary they recognize (commerce, architecture, agents, harness)
2. Within 30 seconds: sees ≥1 named client OR ≥1 named shipped commerce product
3. Within 60 seconds: can reach Cal.com booking flow OR a structured contact path in ≤2 clicks
4. Primary CTA ("Schedule a conversation") is always visible (sticky or persistent in header)

**Anti-acceptance**:
- CTA buried mid-page with no button treatment
- Capability statement reads as brand line ("Cut the Noise, Follow the Signal") instead of capability statement
- Contact path requires scrolling through 3+ sections to find
- No named clients visible; receipts read as personal projects, not commerce credibility

## J3 — Conference Circuit curator slates Nino

| | |
|---|---|
| **Surface** | `/speaking` |
| **Persona** | P3 Conference Circuit (`personas/conference-circuit.md`) |
| **Trigger** | Direct navigation from search OR referral with intent to evaluate for speaking slot |
| **Time budget** | 2 minutes total (30s disqualify, 90s shortlist, 8min deep-dive if seriously considering) |
| **Job** | Slate Nino into a speaker doc without leaving the site |

**Acceptance**:
1. Within 30 seconds: sees three bio variants (50/100/250 word), copy-pasteable with one click each
2. Within 60 seconds: can download at least one headshot (square + wide; with download link, not "ask for it")
3. Within 90 seconds: sees ≥5 named topics with format hints (keynote duration, panel vs workshop)
4. By 2 minutes: can send scoped inquiry with subject template pre-filled, OR copy bio for their slate doc

**Anti-acceptance**:
- Bio shown as a single 250-word paragraph (curator has to manually extract a 50-word version)
- Headshot section says "contact for headshot" instead of providing direct download
- Topics listed without format hints (curator can't tell if it's a keynote, panel, or workshop)
- Inquiry path is generic mailto with no subject template (low-information email exchange follows)

## J4 — Returning Peer Architect checks what's current

| | |
|---|---|
| **Surface** | `/now` |
| **Persona** | P1 Peer Architect (returning, prior visit) |
| **Trigger** | Direct URL OR header link from any page |
| **Time budget** | 30 seconds — quick freshness check |
| **Job** | See what's changed since last visit; decide whether to read a new post or re-engage |

**Acceptance**:
1. Within 5 seconds: sees "last updated" timestamp prominent enough to scan
2. Within 15 seconds: sees ≥3 named in-flight initiatives with stage status (e.g., "Rally HQ — production," "v3 — shipped Stage 6")
3. Within 30 seconds: sees ≥3 recent Signal Dispatch post titles with dates
4. RSS subscribe link visible on the page

**Anti-acceptance**:
- "Last updated" buried in footer or absent
- In-flight section is aspirational ("things I'm thinking about") instead of state-anchored ("Rally HQ: production, 17 tournaments this weekend")
- Recent dispatches section uses placeholder text instead of live RSS

## J5 — Visitor evaluates the methodology behind the practice

| | |
|---|---|
| **Surface** | `/colophon` |
| **Persona** | P1 Peer Architect (deep curiosity) OR P2 Prospective Client (vetting methodology engagement) |
| **Trigger** | Click from footer "Built with Blueprint methodology" OR direct nav |
| **Time budget** | 3 minutes — willing to read |
| **Job** | Verify the methodology is real (not marketing); assess whether it's borrowable / hireable |

**Acceptance**:
1. Within 15 seconds: sees the seven-stage Blueprint pipeline with concrete artifact references (file paths + line counts)
2. Within 60 seconds: sees what changed v2 → v3 with diagnosis IDs (D1, D5, etc.) — provable, not aspirational
3. Within 2 minutes: sees stack details (SvelteKit, Cloudflare, Vectorize, font choices) + predecessors/influences cited by name
4. Can navigate to the actual research artifacts in the repo via working GitHub links

**Anti-acceptance**:
- Methodology described in prose paragraphs without concrete file references
- "What changed from v2" framed as a marketing pitch instead of a closed-diagnosis table
- Predecessors section absent or generic ("inspired by great designers")
- No working repo links — claims are unverifiable

---

## Cross-cutting acceptance (applies to all pages)

- **Information density above the fold > 0**: At least one piece of substantive content (receipt, post, status) must be visible before scrolling. No page may load with hero-only content above the fold.
- **Primary CTA accessibility**: "Schedule a conversation" reachable in ≤2 clicks from any page.
- **Real content over placeholders**: No "PHOTO ANCHOR" / "Latest from Signal Dispatch loading…" / "[draft]" placeholder text on production-deployed pages. If real content isn't available, ship the section without it instead of with a placeholder.
- **Navigation is navigation**: nav items link to actual pages or to clearly-scoped sections. In-page anchors (`#writing`) get a different visual treatment from real routes (`/speaking`) so users know which is which.

  function; explicit Stage 2.5 may be optional
